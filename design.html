Arrow: User Observation

The client wanted a device to aid him in the retrieval of objects from underneath furniture. When playing with his puppy, often times tennis balls would roll underneath the coach or dressers and retrieval would become difficult.

The client would lie flat on the ground and use a yard stick to retrieve the tennis balls.

Many pieces of furniture in the client's house had space for the tennis balls to roll under.

"Recognizing the need is the primary condition for design."

Pass

The client wanted a device to aid in the rehabiliation of stroke patients who were relearning to walk. A common side-effect of a stroke is the loss of certain muscle control which can often times lead to one walking with a "scissoring" type gait. In other words, the patient would cross their legs as they walked.

We worked with the Rehabilitation Institute of Chicago.

Before beginning to brainstorm and design, we met with the client to talk about the exact specifications of the device and find out more about the problem.

Mock-up 1: Magnetic Repullsion

Magnets repelled the legs from one another, increasing resistance as the legs got closer together.

Mock-up 2: Telescoping Tubes

Three connected tubes restricted motion in the undesired directions while allowing the user to move with a natural gait.

Mock-up 3: The Sliding
Belt

A slider attached to a belt increased resistance as the attached leg moved away from its resting position.

The Spec

Some design challenges involve a client who has a very well-defined idea of what he wants. The Portfolio Manager was developed for a Databases course assignment.

Portfolio Manager Assignment

Before beginning work on the project, a clear separation and definition must be achieved for the different levels of the application. The user facing level, the database that exists on the server portion of the application, and the level that connects the two.

Bootstrap was used to aid in the user facing design along with Perl and SQL for the middle and backend logic tiers respectively.

John Lake

Front End: Storyboard

Before coding up the front end of the website, one must have an idea of what the site will look and function like. The storyboard is a sketch that is as close to the final design as possible.

Storyboard

Middle: Flowchart

Underneath the front end website framework, the logic and flow of the functionality must be mapped out. The flowchart shows what links go where and the functionaliy of different pages.

Flowchart

Back End: E/R Diagram

Before implementing the database, theres needs to be an understanding about how data and data relations will be represented. The E/R diagram solidifies the understanding of how the database will represent the data.

E/R Diagram

Front End: Website Mockup

Once the storyboard was complete, a mockup of the website was made, complete with navigation and the general look and layout of the desired final product. The functionality of the website was slowly added once the mockup was done.

Middle: The Logic Teir

The flowchart leads into the design and implmentation of the logic portion of the application. This section of the application was written in Perl and performed various tasks such as data computation and data retrieval given inputs from the user.

Back End: SQL Implementation

The database's portion of the application was written using Oracle SQL. This defined data representation, data relations, and usages for the data. The SQL implementation mirrored the relations set at by the E/R diagram as closely as possible.

Testing

Once the initial prototype is complete, testing needs to be done to in sure everything functions as expected and the users usage patterns are also as expected. This is an iterative process that requires a lot of tuning and testing.

Shipment

In this case, shipment represented a complete portfolio of all the work done during the design process from beginning to the end along with the final working web application. Organization and documentation are key during the design process.
